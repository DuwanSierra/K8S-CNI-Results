% ── 1.2 Pipeline ─────────────────────────────────────────────────────────────
\subsubsection{Pipeline de Aprovisionamiento en Cuatro Etapas}

El repositorio \texttt{K8s-bootstrap} centraliza el despliegue mediante un
único punto de entrada (\texttt{00.bootstrap/}) que compone cuatro módulos
Terraform en cadena. Dos archivos marcadores de estado
(\texttt{.kubeconfig\_ready} y \texttt{.argocd\_ready}) actúan como
barreras: cada etapa posterior solo se inicializa cuando la barrera de la
anterior existe en el sistema de archivos local, haciendo el proceso
seguro ante interrupciones y re-ejecutable sin efectos secundarios.
La Figura~\ref{fig:apa_pipeline} ilustra la secuencia completa.

\begin{verbatim}

|-- 00.bootstrap/                # Orquestador: entrada unica (make apply)
|   |-- main.tf                  # Composicion de los cuatro modulos
|   |-- variables.tf             # Parametros: CNI, region, node sizes
|   `-- scripts/
|       `-- fetch-kubeconfig.sh  # Extrae kubeconfig via SSH con reintentos
|-- 10.k3s-ha-do/                # Etapa 1: Cluster K3s HA
|   `-- main.tf                  # Llama al modulo HA-K3s-Terraform-DO
|-- 20.argo-cd-install/          # Etapa 2: ArgoCD (Helm)
|   `-- main.tf
|-- 30.apps-of-apps-install/     # Etapa 3: ArgoCD Applications (GitOps)

\medskip\noindent\textbf{Condiciones de validez experimental.}
Los pods cliente usan \texttt{podAntiAffinity} preferente respecto
al pod servidor iperf, forzando tráfico \emph{inter-nodo} para
capturar el overhead real del plano de datos del CNI. Cada ejecución
exporta un JSON normalizado con campos \texttt{timestamp\_utc},
\texttt{cni\_name} y métricas resumidas; los runs más antiguos se
eliminan automáticamente manteniendo una retención máxima de
\textbf{5 corridas} por CNI y por tipo de prueba.

// Limpieza de outliers mediante Rango Intercuartilico (IQR)
function filterOutliers(data, key) {
  const values = data.map(d => d[key]).sort((a, b) => a - b);
  const q1 = values[Math.floor(values.length * 0.25)];
  const q3 = values[Math.floor(values.length * 0.75)];
  const iqr = q3 - q1;
  const lowerBound = q1 - 1.5 * iqr;
  const upperBound = q3 + 1.5 * iqr;

  return data.filter(d => d[key] >= lowerBound && d[key] <= upperBound);
}
\end{lstlisting}

\subsection{Objetivo Específico 2: Políticas de Red y Microsegmentación Zero
Trust}

El firewall lógico distribuido se implementó mediante tres casos de
uso de políticas de red declarados como manifiestos Kustomize en
\texttt{K8s-Tesis-Apps} y sincronizados automáticamente por ArgoCD
desde el módulo \texttt{30.apps-of-apps-install} del orquestador.
La variable \texttt{cni\_np\_use\_case} del pipeline selecciona el
overlay activo; Flannel queda excluido por no implementar la API
\texttt{networking.k8s.io/v1}.

\begin{itemize}
    \item \textbf{Repositorio GitOps de Políticas de Red:}
          \url{https://github.com/DuwanSierra/K8s-Tesis-Apps}

    \item \textbf{Estructura de casos de uso (Kustomize):}
\begin{verbatim}
network_policies/
|-- base/                   # Namespace, demo app 3-tier, CronJobs NP
|-- use-case-1-zero-trust/  # deny-all (Ingress+Egress) + allow iperf
|-- use-case-2-multi-tier/  # Segmentacion frontend -> backend -> BD
|-- use-case-3-egress-block/# Bloqueo egress externo, DNS permitido
`-- overlays/
    |-- calico/             # Sin diferenciador propio
    |-- cilium/             # multi-tier: CiliumNetworkPolicy L7 HTTP
    `-- antrea/             # multi-tier: AntreaNetworkPolicy con Tiers
\end{verbatim}

    \item \textbf{Manifiesto de aislamiento base (Caso 1 — Zero Trust):}
    Archivo \path{network_policies/use-case-1-zero-trust/np-deny-all.yaml}
    — declara una \texttt{NetworkPolicy} con \texttt{podSelector: \{\}}
    que bloquea todo ingreso y egreso por defecto en el namespace
    \texttt{cni-np-test}, estableciendo el principio de mínimo
    privilegio como línea base de medición.

    \item \textbf{Política granular L7 en Cilium (Caso 2 — Multi-Tier):}
    Archivo \path{network_policies/overlays/cilium/multi-tier/cilium-l7-policy.yaml}
    — extiende el caso base con una \texttt{CiliumNetworkPolicy} que
    restringe el tráfico HTTP entre tiers al método
    \texttt{GET}/\texttt{HEAD} (inspección L7 habilitada por el
    datapath eBPF de Cilium, exclusiva de este plugin).

    \item \textbf{Activación mediante ArgoCD Application:}
    El caso de uso activo se controla desde el orquestador; el path
    de Kustomize se construye como
    \texttt{network\_policies/overlays/\$\{cni\_provider\}/\$\{use-case\}}.
    Ver archivo
    \path{30.apps-of-apps-install/main.tf} en el repositorio
    \url{https://github.com/DuwanSierra/K8s-bootstrap},
    recurso \texttt{kubernetes\_manifest.cni\_np\_test\_app}.
\end{itemize}

% ─── OE3 ─────────────────────────────────────────────────────────────────────
\subsection{Objetivo Específico 3: Implementación del Algoritmo MCDA}

El algoritmo de decisión multicriterio realiza la normalización de
datos e introduce la penalización dura para garantizar la exclusión
de plugins no seguros.

\begin{itemize}
    \item \textbf{Archivo Clave:}
          \path{cni-recommender-spa/src/data/recommendationModel.js}
          en \url{https://github.com/DuwanSierra/K8S-CNI-Results}
    \item \textbf{Función de Normalización y Scoring MCDA:}
\begin{lstlisting}
// Calculo del ranking MCDA con penalizacion no compensatoria
export function calculateScores(profile, benchmarkPayload) {
  const cnis = benchmarkPayload
    ? buildCniMetricsFromBenchmark(benchmarkPayload) : cniMetrics;
  const securityNeed = profile?.securityNeed ?? 0;
  const securityIsRequired = securityNeed >= 4; // Umbral de restriccion

  return cnis.map((cni) => {
    const rawTotal = Object.entries(profile.weights).reduce(
      (sum, [metric, weight]) => {
        const metricValue = cni.scores[metric] ?? 0;
        return sum + metricValue * weight;
      }, 0);

    // Si requiere seguridad y el CNI no la soporta, penalizar 60%
    const securityPenalized =
      securityIsRequired && cni.supportsNetworkPolicy === false;
    const finalScore = securityPenalized ? rawTotal * 0.4 : rawTotal;

    return {
      ...cni,
      score: Number(finalScore.toFixed(3)),
      rawScore: Number(rawTotal.toFixed(3)),
      securityPenalized,
    };
  }).sort((a, b) => b.score - a.score);
}
\end{lstlisting}
\end{itemize}

\subsubsection*{Integración de datos de laboratorio en tiempo real}

El listado~\ref{lst:fetch_spa} muestra el mecanismo de fetch con
fallback que distingue al prototipo de un recomendador estático.
Si el JSON no está disponible, la UI permanece funcional usando el
conjunto de métricas embebidas; el indicador de estado informa al
operador de la fuente activa.

\begin{lstlisting}[language=Java, caption={Fetch de datos de laboratorio con fallback embebido (\texttt{App.jsx}).}, label={lst:fetch_spa}]
useEffect(() => {
  let isMounted = true;
  fetch(`${import.meta.env.BASE_URL}cni-data.json`,
        { cache: 'no-store' })
    .then((res) => {
      if (!res.ok) throw new Error(`HTTP ${res.status}`);
      return res.json();
    })
    .then((payload) => {
      if (!isMounted) return;
      setBenchmarkData(payload);
      setDataStatus('Datos reales de laboratorio');
    })
    .catch(() => {
      if (!isMounted) return;
      setBenchmarkData(null);
      setDataStatus('Datos de referencia');
    });
  return () => { isMounted = false; };
}, []);
\end{lstlisting}

\section{Guía de Reproducción y Verificación Científica}
\label{sec:apa_reproducibilidad}

Para compilar, validar y reproducir el entorno y el recomendador de
forma local, siga la secuencia de comandos detallada a continuación.

% ── Paso 0 ────────────────────────────────────────────────────────────────────
\subsection{Paso 0: Aprovisionamiento del Testbed}

Para reproducir el entorno de experimentación completo, clonar el
repositorio orquestador e inicializar el pipeline Terraform. El
parámetro \texttt{cni\_provider} acepta los valores
\texttt{flannel}, \texttt{calico}, \texttt{cilium} o \texttt{antrea};
cada valor produce una corrida experimental independiente.

\begin{lstlisting}[language=bash]
# Clonar el orquestador del testbed
git clone https://github.com/DuwanSierra/K8s-bootstrap.git
cd K8s-bootstrap/00.bootstrap

# Crear terraform.tfvars con los parametros obligatorios:
#   do_token, ssh_key_name, ssh_private_key_path,
#   argocd_admin_password, argocd_root_repo_url,
#   cni_provider (flannel | calico | cilium | antrea),
#   cni_results_git_user, cni_results_git_token, agent_size

# Inicializar providers de Terraform
make init

# Verificar el plan de recursos a crear
make plan

# Aprovisionamiento completo: K3s + ArgoCD + Apps + Secrets (~15 min)
make apply

# Desmontaje completo del entorno (limpia state y droplets)
make destroy
\end{lstlisting}

\noindent Al finalizar \texttt{make apply}, los CronJobs de benchmark
comenzarán a ejecutarse automáticamente vía sincronización GitOps de
ArgoCD. Los resultados se publican en
\path{K8S-CNI-Results/results/cni-benchmarks/} cada 30 minutos.

% ── Paso 1 ────────────────────────────────────────────────────────────────────
\subsection{Paso 1: Clonación del Repositorio y Reconstrucción de Datos}

Clonar el repositorio principal del proyecto y ejecutar el procesador
automatizado de Node.js:

\begin{lstlisting}[language=bash]
# Clonar el proyecto
git clone https://github.com/DuwanSierra/K8S-CNI-Results.git
cd K8S-CNI-Results

# Instalar dependencias del pipeline y procesar datos brutos
node docs/procesador.js
\end{lstlisting}

Este comando procesará los datos brutos guardados bajo
\path{results/cni-benchmarks/}, calculará la limpieza IQR de outliers
y generará el archivo unificado en
\path{results/processed/thesis_data.json}.

% ── Paso 2 ────────────────────────────────────────────────────────────────────
\subsection{Paso 2: Compilación y Despliegue de la SPA Recomendadora}

Navegar al directorio de la aplicación web, instalar dependencias de
desarrollo local y levantar el servidor web:

\begin{lstlisting}[language=bash]
# Cambiar al directorio del recomendador web
cd cni-recommender-spa

# Instalar dependencias de React
npm install

# Lanzar el servidor de desarrollo local
npm run dev
\end{lstlisting}

El servidor local se desplegará en \url{http://localhost:5173/},
permitiendo interactuar con los deslizadores y replicar los perfiles
Fintech, Streaming e IoT descritos en la investigación.
