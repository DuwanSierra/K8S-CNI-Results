\chapter{Recomendaciones} \label{sec:Recomendaciones}

A partir de la evidencia técnica recolectada, se formulan las siguientes recomendaciones para la arquitectura y operación de clústeres Kubernetes:

\begin{itemize}
    \item \textbf{Evitar Flannel en entornos con requisitos de seguridad:} Dado que Flannel carece de soporte para aplicar Network Policies, su uso en producción debe prohibirse si la arquitectura requiere aislamiento entre namespaces o pods.
    
    \item \textbf{Implementar Cilium para microsegmentación L7:} En clústeres que soporten cargas transaccionales o APIs críticas (perfil Fintech), se recomienda adoptar Cilium para habilitar el filtrado eBPF de protocolos de capa de aplicación (HTTP/gRPC) sin requerir el despliegue paralelo de un Service Mesh.
    
    \item \textbf{Priorizar Flannel en hardware con limitaciones severas:} Para nodos de Edge computing o IoT, Flannel ofrece la mayor eficiencia base, consumiendo solo un 19.45\% de CPU y 1176.54 MB de RAM bajo estrés de red, en comparación con los requerimientos de Calico (1672.83 MB) o Cilium (1497.19 MB).
    
    \item \textbf{Auditar empíricamente el \emph{enforcement} de red:} Las configuraciones de seguridad no deben darse por válidas únicamente consultando el estado de la API de Kubernetes. Es necesario incorporar rutinas de prueba activa de conectividad TCP/UDP para confirmar que el datapath del CNI aplica efectivamente las reglas.
    
    \item \textbf{Declarar infraestructura vía GitOps:} La instalación del CNI y sus políticas de red asociadas deben estar versionadas en Git y sincronizadas a través de controladores como ArgoCD. Esto previene derivas de configuración (configuration drift) y asegura que las reglas de seguridad sean reproducibles ante desastres.
\end{itemize}
